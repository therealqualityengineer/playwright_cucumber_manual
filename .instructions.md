# Playwright Cucumber Test Framework — Design Pattern Instructions

You are working in a test automation framework built on **Playwright + Cucumber + TypeScript**. All new tests, pages, and step definitions **must follow the established design patterns** documented in `CLAUDE.md`. This file reinforces those patterns and provides a checklist for development.

---

## Core Design Principles

### 1. **Page Objects Pattern**

- **All page classes** extend `BasePage` (located in `pages/BasePage.ts`)
- **Constructor**: accepts `Page`, passes to `super(page)`
- **Selectors**: use **only** `getByRole()` and `getByPlaceholder()` — never CSS selectors or XPath
- **Navigation**: use `this.navigateTo(path)` (inherited from `BasePage`) — do not manually construct URLs
- **Private `fillField(field, value)` method**: maps DataTable field names to locators via a `switch` statement; throw on unknown fields

**Example structure:**

```typescript
import { Page } from "@playwright/test";
import BasePage from "./BasePage";

// Note: DEFAULT_*_DETAILS is typically NOT exported — used only locally in the page class

export interface WidgetDetails {
  widgetId: string; // required identity field
  widgetName?: string; // optional defaultable field
  widgetType?: string;
}

const DEFAULT_WIDGET_DETAILS: Required<Omit<WidgetDetails, "widgetId">> = {
  widgetName: "Default Widget",
  widgetType: "Standard",
};

export default class WidgetPage extends BasePage {
  constructor(page: Page) {
    super(page);
  }

  async navigateToWidget() {
    await this.navigateTo("/wfportal/widget.cfm");
  }

  async createWidget(details: WidgetDetails) {
    const merged = { ...DEFAULT_WIDGET_DETAILS, ...details };
    await this.fillField("widgetName", merged.widgetName);
    await this.fillField("widgetType", merged.widgetType);
    await this.page.getByRole("button", { name: /create/i }).click();
  }

  private async fillField(field: string, value: string) {
    switch (field) {
      case "widgetName":
        await this.page.getByPlaceholder("Widget Name").fill(value);
        break;
      case "widgetType":
        await this.page
          .getByRole("combobox", { name: /type/i })
          .selectOption(value);
        break;
      default:
        throw new Error(`Unknown field: ${field}`);
    }
  }

  async waitForWidgetId(widgetId: string) {
    await this.page.waitForURL(`**/widget.cfm?id=${widgetId}`);
    return widgetId;
  }
}
```

### 2. **Step Definitions**

- **One step definition file per feature file** (e.g., `Widget.feature` → `Widget.steps.ts`)
- **Access page objects** via `this.widgetPage` (from `CustomWorld`)
- **Dynamic placeholder resolution** — implement before passing to page methods:
  - `<RandomAlphabets>` → `RandomAlphabets()`
  - `<RandomNumbers>` → `RandomNumbers()`
  - `<RandomEmail>` → `RandomEmail()`
  - `<RandomString>` → `RandomString()`
  - `<Today>`, `<Today+N>`, `<Today-N>` → `ResolveDate(placeholder)`
  - `<this.fieldName>` → replace with value from `CustomWorld` at runtime
- **Capture scenario state** on `CustomWorld` (e.g., `this.widgetId`, `this.widgetName`) for use in subsequent steps
- **Build DataTable** as `Partial<WidgetDetails>`, cast to `WidgetDetails` before calling page methods

**Example structure:**

```typescript
import { DataTable, Given, Then, When } from "@cucumber/cucumber";
import { expect } from "@playwright/test";
import {
  RandomString,
  RandomEmail,
  ResolveDate,
} from "../test-data/ResolveDynamicData";
import { CustomWorld } from "../utils/CustomWorld";
import { WidgetDetails } from "../pages/WidgetPage";

Given(
  "the user navigates to the Widget page",
  async function (this: CustomWorld) {
    await this.widgetPage.navigateToWidget();
  },
);

When(
  "the user creates a widget with the following details",
  async function (this: CustomWorld, dataTable: DataTable) {
    const row = dataTable.rowsHash();

    // Resolve placeholders
    const widgetName = row["Widget Name"].includes("<RandomString>")
      ? RandomString()
      : row["Widget Name"];

    const details: WidgetDetails = {
      widgetId: "AUTO_ASSIGNED", // will be captured after creation
      widgetName,
      widgetType: row["Widget Type"],
    };

    await this.widgetPage.createWidget(details);

    // Capture ID for later use
    const widgetId =
      await this.widgetPage.waitForWidgetId(/* capture from URL or response */);
    this.widgetId = widgetId;
    this.widgetName = widgetName;
  },
);

Then(
  "the widget should be created successfully",
  async function (this: CustomWorld) {
    expect(this.widgetId).toBeTruthy();
  },
);
```

### 3. **CustomWorld — Page Objects & Scenario State**

- **Two interfaces**: `PageObjects` (browser + page classes) and `ScenarioState` (captured values)
- **TypeScript constraint**: `class CustomWorld implements PageObjects, ScenarioState` ensures compile-time drift detection
- **Add new page**: add property to `PageObjects` interface, declare with `!`, instantiate in `Before` hook
- **Add new state**: add optional property to `ScenarioState` interface

**Example structure:**

```typescript
import { World } from "@cucumber/cucumber";
import { BrowserContext, Page } from "@playwright/test";
import WidgetPage from "../pages/WidgetPage";
import ClientPage from "../pages/ClientPage";

export interface PageObjects {
  context: BrowserContext;
  page: Page;
  widgetPage: WidgetPage;
  clientPage: ClientPage;
}

export interface ScenarioState {
  widgetId?: string;
  widgetName?: string;
  clientId?: string;
  clientName?: string;
}

export class CustomWorld extends World implements PageObjects, ScenarioState {
  context!: BrowserContext;
  page!: Page;
  widgetPage!: WidgetPage;
  clientPage!: ClientPage;

  widgetId?: string;
  widgetName?: string;
  clientId?: string;
  clientName?: string;
}
```

### 4. **Browser Lifecycle (Hooks)**

- **`BeforeAll`**: launch single Chromium browser (non-headless, shared across all scenarios)
- **`Before` (each scenario)**:
  - Create new `BrowserContext` + `Page`
  - Instantiate all page objects
  - Start Playwright tracing
- **`After` (each scenario)**:
  - On failure: attach full-page screenshot + save trace zip to `allure-results/traces/`
  - Always close context
- **`AfterAll`**: close browser, clean up `downloads/`

**Do not deviate** from this lifecycle — it ensures browser isolation and proper cleanup.

### 5. **API Testing Pattern**

- **Single page class**: `APItestPage` with generic `callApi(method, params)` — handles GET and POST
- **Base path**: `/wfportal/clearConnect/2_0/` (auto-prepended)
- **`action` parameter**: auto-injected from step text; never include in DataTable
- **Registry**: `API_METHOD_MAP` in `APItest.steps.ts` maps action name → HTTP method; throw if not registered
  - Current entries: `getTemps: 'GET'`, `getClients: 'GET'`
  - **When adding new API**: register in `API_METHOD_MAP` and add placeholder resolvers in both `Given` and `Then` steps
- **Response handling**: top-level array → use `[0]`; object with `rows` → use `rows[0]`; plain object → use as-is
- **Auth**: Basic header (resolved from `apiConfig` per environment)
- **Placeholders** (supported in both call and assertion steps):
  - `<this.tempId>`, `<this.tempFirstName>`, `<this.tempEmail>`
  - `<this.clientId>`, `<this.clientName>`
  - Add new placeholders by updating resolver logic in `APItest.steps.ts` and declaring on `ScenarioState`

**Adding new API action:**

1. Register in `API_METHOD_MAP`: `{ actionName: 'GET' }`
2. If action uses new `<this.*>` placeholder, add resolver in both `Given` and `Then` steps, declare on `ScenarioState`
3. Write feature scenario with exact response field names in assertions

### 6. **Folder Structure (Fixed)**

```
features/
  feature/        # One .feature file per domain entity
  stepDef/        # One .steps.ts file per feature file
hooks/
  hooks.ts        # BeforeAll/AfterAll/Before/After lifecycle
pages/
  BasePage.ts     # Abstract base (do not modify structure)
  *Page.ts        # One class per domain entity
test-data/
  env-Data.ts     # Environment → URL map
  ApplicationUrls.ts  # App-relative paths
  apiConfig.ts    # API base URLs + credentials
  users.json      # Credentials keyed by username
  ResolveDynamicData.ts  # Random generators + date resolver
utils/
  CustomWorld.ts  # Cucumber World + state interfaces
reports/        # HTML reports (auto-generated)
allure-results/ # Allure results (auto-generated)
allure-report/  # Allure HTML (auto-generated)
```

**New page files go in `pages/`; new feature/step files go in `features/feature/` and `features/stepDef/`** — do not create subdirectories.

### 7. **Tags for Organization**

Always tag scenarios with one or more of:

- `@smoke` — critical path, fast subset
- `@regression` — full regression suite
- `@api` — API-level tests via ClearConnect
- `@<ticketId>` — ticket-level targeting (e.g., `@23455`)

```gherkin
@regression @api @23455
Scenario: Create widget via API and verify
  Given the user perform 'createWidget' API call with the following details
    | fieldName | value        |
    | name      | My Widget    |
```

### 8. **Selectors — Always Use `getByRole` / `getByPlaceholder`**

❌ **NEVER**:

- CSS selectors: `.button-primary`, `#submit-btn`
- XPath: `//button[@type='submit']`
- Implicit selectors: `.page > div > button`

✅ **ALWAYS**:

- `page.getByRole('button', { name: /submit/i })`
- `page.getByPlaceholder('Enter email')`
- `page.getByLabel('Agree to terms')`
- `page.getByText('Create Widget')`

This ensures resilience to CSS changes and improves readability.

---

### 9. **Async/Await — Enforced by ESLint**

- **All async calls must be awaited** (ESLint rule: `@typescript-eslint/no-floating-promises`)
- **All Playwright operations are async** — always `await`

❌ **WRONG**:

```typescript
this.page.click("button"); // missing await
this.page.fill("input", "value"); // missing await
```

✅ **CORRECT**:

```typescript
await this.page.getByRole("button").click();
await this.page.getByPlaceholder("Email").fill("user@example.com");
```

### 10. **Dynamic Test Data**

Use `ResolveDynamicData.ts` functions for dynamic values:

```typescript
import {
  RandomAlphabets,
  RandomNumbers,
  RandomEmail,
  RandomString,
  ResolveDate,
} from "../test-data/ResolveDynamicData";

// In step definitions:
const name = RandomString(); // e.g., "xYzAbC"
const email = RandomEmail(); // e.g., "test_12345@example.com"
const dob = ResolveDate("<Today-18>"); // 18 days ago, MM/DD/YYYY
```

**In DataTable**, use placeholders:

```gherkin
When the user creates a widget with the following details
  | Widget Name | <RandomString>         |
  | Widget Type | Standard               |
  | Created On  | <Today>                |
  | Expires On  | <Today+30>             |
```

**Field naming**: DataTable field names must match the interface property names exactly (e.g., `ClientName` for `ClientDetails.ClientName`). Mismatch will throw `Unknown field` error in `fillField()`.

Then resolve **before** passing to page:

```typescript
const name = row["Widget Name"].includes("<RandomString>")
  ? RandomString()
  : row["Widget Name"];
```

## PR / Review Checklist

When creating **new tests**, validate:

- [ ] **Page Object**: extends `BasePage`, uses only `getByRole`/`getByPlaceholder`
- [ ] **`DEFAULT_*_DETAILS`**: includes all optional fields with sensible defaults; typically NOT exported
- [ ] **`fillField()` switch**: has case for every field in the interface; field names must match DataTable exactly
- [ ] **Step definitions**: one file per feature, imports `CustomWorld` explicitly
- [ ] **Dynamic data**: placeholders resolved before passing to page methods
- [ ] **Scenario state**: new captures stored on `this` (CustomWorld)
- [ ] **Tags**: at least one of `@smoke`, `@regression`, `@api`, or `@ticketId`
- [ ] **Async/await**: all async calls have `await` (ESLint will enforce)
- [ ] **API actions**: registered in `API_METHOD_MAP` if new; add placeholder resolvers to both `Given` and `Then` steps
- [ ] **CustomWorld**: new page type added to `PageObjects` interface, new state to `ScenarioState`
- [ ] **Hooks**: page instantiated in `Before` hook if it's a new page
